## Challenge Description

> A small company moved some of its resources to a private cloud installation. The developers left the redirect to the dashboard on their web server. The security team got an email from the alleged attacker stating that the user data was leaked. It is up to you to investigate the situation.

**Difficulty:** Very Easy  
**Category:** DFIR / Network Forensics  
**Evidence Files:** `web-server_2025-07-01.pcap` (15 MB), `controller_2025-07-01.pcap` (6.9 MB)

## Initial Analysis Methodology

Before diving into the challenge questions, the goal is to understand what happened using only the evidence. This section walks through a repeatable approach you can apply to any unknown PCAP — the same steps, in the same order, every time.

We have two PCAP files from two different machines. We don't know yet what those machines are, how they relate to each other, or what happened. The methodology below will answer all three.

---

### Step 1 — Protocol Hierarchy: What kind of machines are these?

The very first thing to do with any unknown PCAP is find out what protocols it contains. This tells you what kind of machine was captured and what kind of activity to expect — before you read a single packet.

**Wireshark:** `Statistics → Protocol Hierarchy`

![Wireshark protocol hierarchy walkthrough](/images/sherlocks/vantage/wireshark-step-1-protocol-hierarchy.svg)

#### 📁 web-server_2025-07-01.pcap

Open the web-server PCAP first and go to `Statistics → Protocol Hierarchy`. Here's what stands out:

| Protocol              | Frames | What it tells you                                      |
| --------------------- | ------ | ------------------------------------------------------ |
| TCP                   | 21,586 | Nearly all traffic is TCP-based                        |
| HTTP                  | 7,482  | A web server — this is almost entirely HTTP            |
| HTML Form URL Encoded | 8      | **Only 8 form submissions out of 7,482 HTTP requests** |

**Why the 8 form submissions matter:** The content type `application/x-www-form-urlencoded` is what browsers send when you submit a login form via POST. Out of 7,482 HTTP requests, only 8 were form POSTs — meaning the vast majority of traffic was automated GETs (scanning/fuzzing), and only a handful were deliberate human interactions (login attempts). That ratio is the first red flag, and you've spotted it without reading a single packet.

#### 📁 controller_2025-07-01.pcap

Now open the controller PCAP and check the same view:

| Protocol           | Frames | What it tells you                  |
| ------------------ | ------ | ---------------------------------- |
| HTTP + JSON        | 626    | REST API traffic                   |
| MySQL              | 8,978  | Database backend                   |
| AMQP               | 849    | Message queue (RabbitMQ)           |
| Data on port 2379  | 1,418  | etcd — distributed key-value store |
| Data on port 11211 | —      | memcached — in-memory cache        |

**Important honesty note:** The protocol hierarchy alone does _not_ tell you "this is OpenStack." HTTP + MySQL + AMQP + etcd + memcached could be many things — a Django app with Celery workers, a Kubernetes control plane, or any message-driven platform with a database. The protocols narrow it to "some kind of application controller node," but that's as far as they go.

**What actually confirms OpenStack** comes later, when you look at the HTTP content (Steps 3–5). Here's the real identification chain, in the order the evidence appears:

1. **Protocol hierarchy (this step):** HTTP + MySQL + AMQP + etcd + memcached on one machine → _"this is an application controller of some kind."_ A useful narrowing, but not a fingerprint.

2. **HTTP URIs (Step 3):** When you filter `http.request` in this PCAP, the paths give it away instantly: `/identity/v3/auth/tokens`, `/compute/v2.1/servers/detail`, `/networking/v2.0/extensions`, `/volume/v3/limits`. These are the canonical OpenStack service API paths — Keystone, Nova, Neutron, Cinder. No other platform uses this exact URL structure.

3. **The web-server PCAP has it in plain text (Step 3):** In the web-server PCAP, one of the HTTP URIs is `/dashboard/i18n/js/horizon+openstack_dashboard/` — this literally contains the words `horizon` (OpenStack's dashboard name) and `openstack_dashboard`. You'd see this just by scrolling through `http.request` packets.

4. **The OpenRC file spells it out (Step 5):** Following the TCP stream on the `/dashboard/project/api_access/openrc/` download reveals a shell script whose comments say _"To use an OpenStack cloud you need to authenticate against the Identity service named keystone."_ The environment variables `OS_AUTH_URL`, `OS_PROJECT_ID`, and `OS_USERNAME` are all OpenStack-specific.

5. **The attacker's User-Agent (Step 4):** The CLI requests in the controller PCAP carry `openstacksdk/4.6.0 keystoneauth1/5.11.1` — the OpenStack Python SDK identifying itself by name.

In short: the protocols give you a _hint_, but the HTTP URIs and file contents give you _certainty_. Don't claim to "fingerprint OpenStack from the protocol hierarchy" — that's overstating what the protocols alone can tell you. The confirmation always comes from reading the actual request paths and payloads.

**The takeaway after Step 1:** We have a _web server_ (almost entirely HTTP) and a _multi-service controller node_ (API + database + message queue + cache). We don't yet know for certain what platform the controller runs — that will become obvious in the next steps when we look at the HTTP traffic. What we do know is that these are two different machines in the same infrastructure, and the web-server PCAP is the one with the suspicious traffic volume.

### Step 2 — Endpoints & Conversations: Who's talking, and how much?

Now we know _what_ these machines are. Next question: _who is communicating with them_, and is any of that traffic abnormal?

There are two related views in Wireshark that answer this differently:

- **`Statistics → Endpoints → IPv4`** — One row per IP, total packets aggregated across all connections. Best for quickly identifying dominant talkers.
- **`Statistics → Conversations → TCP`** — One row per TCP stream (unique source port ↔ destination port pair). Best for understanding _how_ the traffic is structured.

**Start with Endpoints for triage, then drill into Conversations for detail.**

![Wireshark endpoints and conversations walkthrough](/images/sherlocks/vantage/wireshark-step-2-conversations.svg)

#### 📁 web-server_2025-07-01.pcap — Endpoints view

Open `Statistics → Endpoints → IPv4 tab` and sort by the **Packets** column (descending):

| Address           | Packets    | % of total | Role                      |
| ----------------- | ---------- | ---------- | ------------------------- |
| 157.230.81.229    | 21,348     | —          | Web server (this machine) |
| **117.200.21.26** | **20,897** | **97.9%**  | **??? (suspicious)**      |
| 10.116.0.3        | 256        | 1.2%       | Internal IP               |
| 10.116.0.4        | 256        | 1.2%       | Internal IP               |
| 60+ other IPs     | 2–243 each | &lt;1%     | Internet background noise |

**The red flag is immediate:** One external IP — `117.200.21.26` — accounts for 97.9% of all traffic. Every other external IP has negligible volume. That ratio alone identifies the attacker before you've looked at a single HTTP request.

#### 📁 web-server_2025-07-01.pcap — Conversations view

Now switch to `Statistics → Conversations → TCP tab` to understand how that traffic is structured. You'll see approximately **187 rows** from `117.200.21.26`, all connecting to `157.230.81.229:80`, each from a different source port:

| Address A                         | ↔   | Address B         | Packets | Duration |
| --------------------------------- | --- | ----------------- | ------- | -------- |
| 117.200.21.26:**64728**           | ↔   | 157.230.81.229:80 | 497     | 22.6s    |
| 117.200.21.26:**64727**           | ↔   | 157.230.81.229:80 | 489     | 22.7s    |
| 117.200.21.26:**64756**           | ↔   | 157.230.81.229:80 | 487     | 22.6s    |
| _... 184 more rows like this ..._ |     |                   |         |          |

**What this pattern means:** 187 parallel TCP connections, all nearly identical in duration (~22 seconds) and packet count (~480–500), all hitting port 80. This is not a human browsing — this is a _tool running concurrent threads_. Each thread opens its own TCP connection to fire HTTP requests in parallel. Tools like ffuf, gobuster, and dirsearch all behave exactly this way.

There are also ~10 smaller streams from `10.116.0.3 → 10.116.0.4:80` with much less traffic (~250 packets total). These two internal IPs have nearly identical packet counts in the Endpoints view, and their timing aligns with the external wave — this is the hallmark of a reverse proxy chain (e.g., Nginx/HAProxy fronting a backend app server).

#### 📁 controller_2025-07-01.pcap — What about this one?

We'll come back to the controller PCAP after we understand the web-server traffic. The methodology is: **follow the attacker's footsteps in order**. The web server is where the attack started; the controller is where it escalated. Analyzing them chronologically tells a coherent story.

### Step 3 — Filter HTTP Requests: What was the attacker doing?

We've identified `117.200.21.26` as suspicious. Now let's see _exactly_ what they requested.

**Wireshark display filter (in `web-server_2025-07-01.pcap`):**

```
http.request && ip.src == 117.200.21.26
```

![Wireshark HTTP filter walkthrough](/images/sherlocks/vantage/wireshark-step-3-filter-http.svg)

**Tip:** To make the packet list more readable, add custom columns. Right-click any column header → `Column Preferences` → add:

- `http.host` (titled "Host")
- `http.request.uri` (titled "URI")
- `http.user_agent` (titled "User-Agent")

With those columns visible, the pattern jumps out immediately:

| #     | Host                        | Method | URI                    |
| ----- | --------------------------- | ------ | ---------------------- |
| 1024  | 1.vantage.tech              | GET    | /                      |
| 1031  | 2.vantage.tech              | GET    | /                      |
| 1038  | a.vantage.tech              | GET    | /                      |
| 1052  | abc.vantage.tech            | GET    | /                      |
| 1059  | about.vantage.tech          | GET    | /                      |
| ...   | _3,579 unique subdomains_   | GET    | /                      |
| 1072  | cloud.vantage.tech          | GET    | /                      |
| —     | _(User-Agent changes here)_ |        |                        |
| 18201 | cloud.vantage.tech          | GET    | /dashboard/            |
| 18215 | cloud.vantage.tech          | GET    | /dashboard/auth/login/ |
| 18240 | cloud.vantage.tech          | POST   | /dashboard/auth/login/ |

**Two distinct phases are visible:**

1. **Fuzzing phase:** Thousands of GETs, each to a different `*.vantage.tech` subdomain, all requesting `/`. The Host header cycles through a wordlist alphabetically. This is subdomain enumeration.
2. **Exploitation phase:** The Host locks onto `cloud.vantage.tech` and the attacker starts browsing the dashboard manually — navigating to the login page, submitting POST requests.

The transition between phases is also visible in the User-Agent column (which we'll examine next).

---

### Step 4 — Check User-Agents: What tools were used?

User-Agent strings are one of the quickest wins in web traffic analysis. Legitimate browsers all send `Mozilla/5.0...` headers. Attack tools often identify themselves — either by name or by sending unusual/missing User-Agents.

**Wireshark (in `web-server_2025-07-01.pcap`):** Click any packet from the fuzzing phase, expand `Hypertext Transfer Protocol → User-Agent` in the packet detail pane.

![Wireshark user-agent inspection walkthrough](/images/sherlocks/vantage/wireshark-step-4-user-agent.svg)

Two User-Agents appear from the attacker's IP:

| Phase                | User-Agent                         | Meaning                                |
| -------------------- | ---------------------------------- | -------------------------------------- |
| Fuzzing (3,579 GETs) | `Fuzz Faster U Fool v2.1.0-dev`    | **ffuf** — a Go-based web fuzzing tool |
| Manual browsing      | `Mozilla/5.0 ... Chrome/137.0.0.0` | Real Chrome browser                    |

**The pivot moment:** The switch from ffuf to Chrome is the exact moment the attacker discovered `cloud.vantage.tech` returned something interesting (an OpenStack Horizon dashboard) and decided to explore it manually.

To filter for just the non-browser traffic:

```
http.user_agent && !(http.user_agent contains "Mozilla")
```

This isolates every request made by tools rather than browsers — useful for quickly spotting automated activity in any PCAP.

---

### Step 5 — Follow TCP Streams: Read the full conversations

So far we've been looking at metadata — source IPs, URIs, User-Agents. Now we need to see the actual _content_ of the most interesting requests: what credentials were submitted, what files were downloaded, what the server responded with.

**Wireshark (in `web-server_2025-07-01.pcap`):** Right-click any login POST packet → `Follow → TCP Stream`

![Wireshark follow TCP stream walkthrough](/images/sherlocks/vantage/wireshark-step-5-follow-stream.svg)

This reconstructs the full HTTP conversation — the complete request (red text) and the complete response (blue text) — in a readable window. For the login POSTs, the form data is visible in cleartext:

```
POST /dashboard/auth/login/ HTTP/1.1
Host: cloud.vantage.tech
Content-Type: application/x-www-form-urlencoded

username=admin&password=StrongAdminSecret&region=default
```

**How to tell success from failure without reading the HTML:** Look at the HTTP response code. For most web login forms:

- **HTTP 200 OK** = the server re-rendered the login page → **login failed**
- **HTTP 302 Found** (redirect to dashboard) → **login succeeded**

You can check this by looking at the response in the TCP stream, or by filtering for responses:

```
http.response && ip.dst == 10.116.0.3
```

The 302 redirect is the one to focus on — that's the successful login.

**Also follow the OpenRC download stream:** Filter for `http.request.uri contains "openrc"`, then Follow TCP Stream on that packet. The response contains the downloaded `admin-openrc.sh` file:

```bash
export OS_AUTH_URL=http://134.209.71.220/identity
export OS_PROJECT_ID=9fb84977ff7c4a0baf0d5dbb57e235c7
export OS_PROJECT_NAME="admin"
export OS_USERNAME="admin"
```

This is the bridge between the two PCAPs: the OpenRC file tells the attacker the controller's IP address (`134.209.71.220`) and how to authenticate to it. Everything that happens in the controller PCAP after this moment is a consequence of this download.

---

### Step 6 — Cross-Reference PCAPs: Connect the timeline

This is where the two PCAP files merge into a single story. The question is: _does the activity in the web-server PCAP lead to activity in the controller PCAP?_

**Method:** Compare timestamps. Open both files and look at when activity ends in one and begins in the other.

![Wireshark timestamp correlation walkthrough](/images/sherlocks/vantage/wireshark-step-6-time-correlation.svg)

#### 📁 web-server_2025-07-01.pcap — Last meaningful action

The attacker's last request is the OpenRC download at **09:40:29 UTC**.

#### 📁 controller_2025-07-01.pcap — First new API activity

Before 09:40:29, the controller PCAP contains API calls triggered by the Horizon dashboard session (when you browse the dashboard UI, it makes backend API calls automatically). These are "noise" — they happen for any logged-in user.

The first _standalone_ API call — not triggered by the dashboard — is a `GET /identity` at **09:41:44 UTC**. This is the OpenStack CLI's initial service discovery request, which happens when you run your first `openstack` command after sourcing the RC file.

**The 75-second gap** (09:40:29 → 09:41:44) is the attacker reading the OpenRC file, opening a terminal, running `source admin-openrc.sh`, entering the password, and executing their first command. The causal chain is clear.

From this point, everything in `controller_2025-07-01.pcap` is the attacker using the OpenStack CLI directly:

**Filter (in `controller_2025-07-01.pcap`):**

```
http.request && !(http.request.uri contains "keepalive") && frame.time >= "Jul 1, 2025 09:41:00"
```

This shows the post-dashboard API calls, excluding the etcd keepalive noise that runs constantly in the background. You'll see:

1. Service and endpoint enumeration (`/identity/v3/services`, `/identity/v3/endpoints`)
2. User listing (`/identity/v3/users`)
3. Server/VM listing (`/compute/v2.1/servers/detail`)
4. Swift container listing (`/v1/AUTH_.../` with `?format=json`)
5. Individual file downloads (`user-details.csv`, `employee-details.csv`)
6. User creation (`POST /identity/v3/users`) and role assignment (`PUT .../roles/...`)

Each of these is a separate `openstack` CLI command the attacker ran, and the timestamps show them executing sequentially — about 30–90 seconds apart — as the attacker explored the environment manually.

---

### Methodology Summary

Every time you open an unknown PCAP, follow this sequence:

1. **Protocol Hierarchy** → Tells you _what kind of machine_ this is
2. **Endpoints** → Tells you _who's suspicious_ (disproportionate traffic volume)
3. **Conversations** → Tells you _how_ the traffic is structured (parallelism, ports)
4. **`http.request` filter** → Tells you _what they did_ (URIs, hosts, methods)
5. **User-Agents** → Tells you _what tools they used_
6. **Follow TCP Streams** → Tells you _the full content_ of interesting exchanges
7. **Cross-reference timestamps** → Connects activity across multiple captures

Each step narrows your focus. You go from "I have 24,000 packets" to a complete narrative of the attack, without needing to know the questions in advance.

## Questions & Answers

### Task 1: What tool did the attacker use to fuzz the web server?

**PCAP:** `web-server_2025-07-01.pcap`  
**Filter:** `http.request && ip.src == 117.200.21.26`  
**Where to look:** Click any packet → expand `Hypertext Transfer Protocol → User-Agent`

The User-Agent field reads `Fuzz Faster U Fool v2.1.0-dev` — this is ffuf, a popular web fuzzing tool written in Go.

> **Answer:** `ffuf@2.1.0-dev`

---

### Task 2: Which subdomain did the attacker discover?

**PCAP:** `web-server_2025-07-01.pcap`  
**Filter:** `http.request && ip.src == 117.200.21.26 && http.user_agent contains "Chrome"`

After the ffuf scan, the attacker switched to Chrome and manually browsed to the discovered subdomain. Every Chrome request targets the same Host header.

> **Answer:** `cloud.vantage.tech`

---

### Task 3: How many login attempts did the attacker make before successfully logging in?

**PCAP:** `web-server_2025-07-01.pcap`  
**Filter:** `http.request.method == POST && http.request.uri contains "/dashboard/auth/login"`

Four POST requests appear. The response codes distinguish success from failure: HTTP 200 means the login page was re-rendered (failed), while HTTP 302 redirect means authentication succeeded. The first three returned 200, the fourth returned 302.

| Attempt      | Credentials               | Response  | Result  |
| ------------ | ------------------------- | --------- | ------- |
| 1 (09:39:16) | admin : admin             | 200 OK    | Failed  |
| 2 (09:39:27) | demo : demo               | 200 OK    | Failed  |
| 3 (09:39:33) | root : root               | 200 OK    | Failed  |
| 4 (09:40:07) | admin : StrongAdminSecret | 302 Found | Success |

> **Answer:** `3`

---

### Task 4: When did the attacker download the OpenStack API remote access config file?

**PCAP:** `web-server_2025-07-01.pcap`  
**Filter:** `http.request.uri contains "openrc"`

The OpenRC file is a shell script that sets environment variables for OpenStack CLI authentication. The attacker downloaded it from `Project → API Access → Download OpenStack RC File` in the Horizon dashboard.

Following the TCP stream (right-click → Follow → TCP Stream) shows the response with `Content-Disposition: attachment; filename="admin-openrc.sh"`, containing the auth URL (`http://134.209.71.220/identity`), the project ID, and the username. This is the file that bridges the two PCAPs — it gave the attacker everything they needed to pivot from the web server to direct API access on the controller.

> **Answer:** `2025-07-01 09:40:29`

---

### Task 5: When did the attacker first interact with the API on the controller node?

**PCAP:** `controller_2025-07-01.pcap`  
**Filter:** `http.request && !(http.request.uri contains "keepalive") && !(http.request.uri contains "dashboard")`

Before 09:40:29, all API calls on the controller were triggered by the Horizon dashboard session (browsing the UI generates backend API calls automatically). After a 75-second gap (the attacker reading and sourcing the OpenRC file), the first standalone API call appears — a `GET /identity` which is the OpenStack CLI's initial service discovery request.

> **Answer:** `2025-07-01 09:41:44`

---

### Task 6: What is the project ID of the default project accessed by the attacker?

**PCAP:** Both — appears in `web-server` (OpenRC file) and throughout `controller` (API URLs)  
**Easiest source:** Follow TCP Stream on the OpenRC download (Task 4)

```bash
export OS_PROJECT_ID=9fb84977ff7c4a0baf0d5dbb57e235c7
```

It also appears embedded in dozens of API URLs throughout the controller PCAP: compute usage queries, Swift storage paths (`/v1/AUTH_9fb84977...`), and networking quota requests.

> **Answer:** `9fb84977ff7c4a0baf0d5dbb57e235c7`

---

### Task 7: Which OpenStack service provides authentication and authorization?

**PCAP:** `controller_2025-07-01.pcap`  
**Filter:** `http.request.uri == "/identity/v3/services"`

Following the TCP stream reveals the full service catalog JSON. The service with type `identity` is **Keystone** — OpenStack's authentication and authorization service. Every API call hitting the `/identity/` path in the controller PCAP is talking to Keystone.

> **Answer:** `Keystone`

---

### Task 8: What is the endpoint URL of the Swift service?

**PCAP:** `controller_2025-07-01.pcap`  
**Filter:** `http.request.uri == "/identity/v3/endpoints"`

The endpoint catalog JSON response lists all service endpoints. The Swift public endpoint is registered as a template:

```text
http://134.209.71.220:8080/v1/AUTH_$(project_id)s
```

When the `$(project_id)s` template variable is resolved with the admin project ID from the OpenRC file, the full URL becomes:

> **Answer:** `http://134.209.71.220:8080/v1/AUTH_9fb84977ff7c4a0baf0d5dbb57e235c7`

---

### Task 9: How many containers were discovered by the attacker?

**PCAP:** `controller_2025-07-01.pcap`  
**Filter:** `http.request.uri == "/v1/AUTH_9fb84977ff7c4a0baf0d5dbb57e235c7?format=json"`

Following the TCP stream shows the Swift account listing response. The header `X-Account-Container-Count: 3` confirms the count, and the JSON body lists all three:

1. `dev-files`
2. `employee-data`
3. `user-data`

> **Answer:** `3`

---

### Task 10: When did the attacker download the sensitive user data file?

**PCAP:** `controller_2025-07-01.pcap`  
**Filter:** `http.request.uri contains "user-details.csv"`

Two packets match: the external request to Swift proxy (port 8080) and the internal proxy-to-object-server forwarding on `/sdb1/`. The timestamp of the first — the attacker's actual request — is the answer.

> **Answer:** `2025-07-01 09:45:23`

---

### Task 11: How many user records are in the sensitive user data file?

**PCAP:** `controller_2025-07-01.pcap`  
**Method:** Follow TCP Stream on the `user-details.csv` download

The response header shows `Content-Type: text/csv` and `Content-Length: 1367`. The file contains a header row (`Full Name,Email,Phone Number`) followed by 28 data records (John Doe through Zoe Lee), each with full name, email address, and phone number — personally identifiable information.

> **Answer:** `28`

---

### Task 12: What is the username of the new user created for persistence?

**PCAP:** `controller_2025-07-01.pcap`  
**Filter:** `http.request.method == POST && http.request.uri == "/identity/v3/users"`

Following the TCP stream reveals the JSON request body:

```json
{
  "user": {
    "name": "jellibean",
    "password": "P@$$word",
    "enabled": true,
    "default_project_id": "9fb84977ff7c4a0baf0d5dbb57e235c7"
  }
}
```

The server responded with `HTTP 201 CREATED`, confirming the account was successfully created. A subsequent PUT request granted this user the admin role on the project.

> **Answer:** `jellibean`

---

### Task 13: What is the password of the new user?

**PCAP:** `controller_2025-07-01.pcap`  
**Method:** Same TCP stream as Task 12 — the password is in the JSON request body in cleartext.

> **Answer:** `P@$$word`

### Task 14: What is the MITRE ATT&CK technique ID for the persistence method?

The attacker created a new user account in a cloud platform (OpenStack Keystone) and granted it administrative privileges. This maps to:

- **Tactic:** Persistence
- **Technique:** T1136 — Create Account
- **Sub-technique:** .003 — Cloud Account

T1136.003 specifically covers adversaries creating accounts in cloud environments (AWS IAM, Azure AD, GCP, OpenStack Keystone) to maintain persistent access, distinguishing it from local account creation (.001) or domain account creation (.002).

> **Answer:** `T1136.003`
